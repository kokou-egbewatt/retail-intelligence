# GlobalCart Retail Intelligence Engine — Roadmap

This roadmap links each item to the **files and folders** that are involved, and explains **what to do in more depth**. For the overall technical picture, see [TECHNICAL_PRESENTATION.md](TECHNICAL_PRESENTATION.md).

---

## Current state (MVP)

- **Single-notebook RAG:** [retail-intelligence.ipynb](../retail-intelligence.ipynb) — embedding, FAISS, BM25, intent, guardrails, Gradio.
- **Modular pipelines:** [pipelines/ingestion/](../pipelines/ingestion/) (clean, ingest task data) → [pipelines/indexing/](../pipelines/indexing/) (FAISS + metadata) → [vector_store/faiss_index/](../vector_store/faiss_index/).
- **Hybrid retrieval + region filter:** Implemented in notebook and expected in [app/rag/](../app/) (e.g. hybrid_search); CLI: [scripts/run_retrieval.py](../scripts/run_retrieval.py).
- **Security:** No internal data in [data/processed/products_clean.csv](../data/processed/products_clean.csv) or index; jailbreak detection and refusal in notebook §10 and (when present) app.rag.
- **Evaluation:** [evaluation/test_queries.py](../evaluation/test_queries.py) — regional integrity, technical precision, policy summary, security (restricted + injection).
- **Frontend:** [frontend/chat_app.py](../frontend/chat_app.py) (Streamlit) → POST /api/chat. **Backend:** [Dockerfile](../Dockerfile) expects [app/](../app/) (FastAPI); `app/` is not yet in the repo.

Without `app/`, Docker build and these scripts fail. |
| **Unify notebook vs app** | [retail-intelligence.ipynb](../retail-intelligence.ipynb) (§7–§12), [app/rag/](../app/) (when present) | **What:** Ensure intent rules, region mapping, guardrail patterns, context builder fields, and RAG prompt are **identical** in the notebook and in `app.rag`. **How:** Extract shared constants (e.g. COUNTRY_TO_REGION, JAILBREAK_PATTERNS, SAFE_FIELDS, RAG_PROMPT) into a single module (e.g. `app/rag/config.py` or `app/rag/constants.py`) and have both the notebook and the app import them, or copy-paste and keep a short “source of truth” note. **Why:** Prevents drift (e.g. new region in app but not notebook, or different refusal message) and ensures evaluation tests reflect production behaviour. |
| **CI for evaluation** | [evaluation/test_queries.py](../evaluation/test_queries.py), [.github/workflows/](../.github/workflows/) (or other CI config) | **What:** Run the evaluation suite in CI on every push/PR. **How:** Add a workflow that (1) sets up Python, (2) installs deps from [requirements.txt](../requirements.txt), (3) runs ingestion + indexing (e.g. `python scripts/run_indexing.py` or run [clean_data.py](../pipelines/ingestion/clean_data.py) + [build_vector_index.py](../pipelines/indexing/build_vector_index.py) with minimal test data), (4) sets `EVAL_MOCK_LLM=1` and runs `python evaluation/test_queries.py`. **Deeper:** With `EVAL_MOCK_LLM=1`, tests use a mock so no OpenRouter key is needed and runs are fast and deterministic. You can later add a separate job that runs with a real LLM (and a secret key) for release checks. **Link:** Create or edit the CI config under `.github/workflows/` (e.g. `test.yml`). |
| **Docker Compose** | [Dockerfile](../Dockerfile), new `docker-compose.yml` at repo root, [frontend/chat_app.py](../frontend/chat_app.py) | **What:** One command to start the API (and optionally the frontend). **How:** Add `docker-compose.yml` that: (1) builds the backend from the [Dockerfile](../Dockerfile), (2) mounts [vector_store/](../vector_store/) and optionally [.env](../.env.example) or env vars, (3) exposes port 8000. Optionally add a second service for Streamlit (or document that the user runs `streamlit run frontend/chat_app.py` locally with `STREAMLIT_CHAT_API_URL=http://localhost:8000`). **Link:** New file [docker-compose.yml](../docker-compose.yml) (to be created). Document in [README.md](../README.md). |
| **Health and readiness** | [app/main.py](../app/) (when present) | **What:** Endpoints for orchestration and load balancers. **How:** In the FastAPI app, add `GET /health` (e.g. returns 200 and `{"status":"ok"}`) and optionally `GET /ready` that returns 200 only if the FAISS index and metadata are loaded (so readiness fails if [vector_store/faiss_index/](../vector_store/faiss_index/) is missing or broken). **Link:** Implement inside the same [app/main.py](../app/) that serves `/api/chat`. |
| **Logging and errors** | [app/rag/](../app/) (pipeline, main) | **What:** Structured logs (e.g. request_id, query, country, intent, retrieval top_k, latency, error type) and consistent HTTP error responses from the chat API. **How:** Use Python `logging` (or structlog) with JSON format; on exception (e.g. index load failure, OpenRouter timeout), return 5xx with a stable error shape (e.g. `{"error":"internal","message":"..."}`) and log the full trace server-side. **Link:** [app/main.py](../app/) and [app/rag/pipeline.py](../app/). |

## Phase 2 — Data and retrieval

| Item | Links | Deeper explanation |
|------|--------|---------------------|
| **More regions** | [frontend/chat_app.py](../frontend/chat_app.py), notebook §7, [app/rag/](../app/) (region mapping), [scripts/generate_retail_dataset.py](../scripts/generate_retail_dataset.py) | **What:** Support more countries and currencies (e.g. Nigeria, South Africa, Kenya, UK, Germany, France, US, Canada). **Where:** (1) Extend the region map (e.g. COUNTRY_TO_REGION) in the notebook and in app.rag so retrieval filters and display use the same names. (2) Add currency mapping (e.g. in [TECHNICAL_PRESENTATION](TECHNICAL_PRESENTATION.md) §4 table) and use it when formatting prices. (3) The [frontend](../frontend/chat_app.py) already lists these countries in the dropdown; ensure the API accepts them and returns correct currency. (4) Optionally extend [generate_retail_dataset.py](../scripts/generate_retail_dataset.py) to generate more regions. **Link:** Same files as above; consider a single [app/rag/config.py](../app/) or `constants.py` for region + currency. |
| **Policy documents** | [data/raw/](../data/raw/), [pipelines/ingestion/](../pipelines/ingestion/), [pipelines/indexing/build_vector_index.py](../pipelines/indexing/build_vector_index.py), [vector_store/faiss_index/](../vector_store/faiss_index/) | **What:** First-class policy ingestion and indexing so policy answers are robust. **How:** (1) Define a policy schema (e.g. policy_id, country, policy_type, text) and either add policy rows to the same clean CSV or a separate `policies_clean.csv`. (2) In [ingestion](../pipelines/ingestion/), add a step (or script) that loads policy sources (CSV, Excel, or a store) and outputs a policy table. (3) In [build_vector_index.py](../pipelines/indexing/build_vector_index.py) (or a separate policy index), embed policy text and add to the same FAISS index with a `doc_type` (or source) in metadata so hierarchical retrieval can put policy first for policy_question intent. (4) Ensure [clean_data.py](../pipelines/ingestion/clean_data.py) or a dedicated policy cleaner never includes internal/legal-only notes in searchable text. **Link:** New or updated files under [pipelines/ingestion/](../pipelines/ingestion/), [pipelines/indexing/](../pipelines/indexing/), and possibly [data/raw/](../data/raw/) or `data/policies/`. |
| **Relevance tuning** | [app/rag/hybrid_search](../app/), notebook §6, [evaluation/test_queries.py](../evaluation/test_queries.py) | **What:** Tune hybrid weights (e.g. 0.6/0.4) and top_k, optionally per intent. **How:** (1) Make vector_weight and bm25_weight configurable (env or config). (2) Run [evaluation/test_queries.py](../evaluation/test_queries.py) and manual spot-checks with different weights. (3) Optionally add retrieval metrics (e.g. relevance labels or NDCG) and a small eval set. (4) For “SKU lookup”, consider boosting BM25 or lowering top_k so exact SKU matches rank first. **Link:** [app/rag/hybrid_search.py](../app/), [retail-intelligence.ipynb](../retail-intelligence.ipynb) §6. |
| **Index refresh** | [scripts/run_indexing.py](../scripts/run_indexing.py), [pipelines/ingestion/](../pipelines/ingestion/), [pipelines/indexing/](../pipelines/indexing/) | **What:** Rebuild the index from clean data (and policies) without manual steps. **How:** [run_indexing.py](../scripts/run_indexing.py) already runs clean + build_vector_index; document it in [README.md](../README.md) as the standard “refresh index” command. Optionally add a small script or Make target that (1) runs [clean_data.py](../pipelines/ingestion/clean_data.py), (2) runs any policy ingestion, (3) runs [build_vector_index.py](../pipelines/indexing/build_vector_index.py). **Link:** [scripts/run_indexing.py](../scripts/run_indexing.py), [README.md](../README.md). |
| **Embedding model options** | [pipelines/indexing/build_vector_index.py](../pipelines/indexing/build_vector_index.py), [app/rag/hybrid_search](../app/) | **What:** Configurable embedding model (e.g. env var) for future multilingual or larger models. **How:** In [build_vector_index.py](../pipelines/indexing/build_vector_index.py), read model name from env (e.g. `EMBEDDING_MODEL`, default `all-MiniLM-L6-v2`) and pass to SentenceTransformer. In the app retriever, load the same model name when building query embeddings. **Link:** [build_vector_index.py](../pipelines/indexing/build_vector_index.py), [.env.example](../.env.example). |

## Phase 3 — Product and UX

| Item | Links | Deeper explanation |
|------|--------|---------------------|
| **Conversation memory** | [app/rag/pipeline](../app/), [app/main.py](../app/) (chat endpoint), [frontend/chat_app.py](../frontend/chat_app.py) | **What:** Multi-turn context (e.g. last N turns or session summary) so the model can refer to “the kettle I asked about”. **How:** (1) API: accept an optional `session_id` or `history` (list of {role, content}); append last N user/assistant turns to the RAG prompt or to a “conversation” block. (2) Retriever: optionally expand the query with a short summary of the previous turn (e.g. “previous: price of Solar Inverter in Ghana”) to improve retrieval. (3) Frontend: keep session_state.messages and send last K turns in the request, or use a backend session store keyed by session_id. **Link:** [app/rag/pipeline.py](../app/), [app/main.py](../app/), [frontend/chat_app.py](../frontend/chat_app.py). |
| **Citation and sources** | [app/rag/pipeline](../app/), [app/main.py](../app/) (response shape), [frontend/chat_app.py](../frontend/chat_app.py) | **What:** Return and display which products/policies were used for each answer. **How:** (1) Pipeline returns not only `response` but also `sources` (e.g. list of {product_id, country, item_name, snippet}). (2) API response body: `{"response":"...", "sources":[...]}`. (3) Frontend renders sources below the message (e.g. “Based on: Product X (Ghana), Policy Y (Netherlands)”). **Link:** [app/rag/pipeline.py](../app/), [app/main.py](../app/), [frontend/chat_app.py](../frontend/chat_app.py). |
| **Streaming responses** | [app/main.py](../app/) (StreamingResponse), [app/rag/pipeline](../app/) (LLM call), [frontend/chat_app.py](../frontend/chat_app.py) | **What:** Stream LLM tokens to the client for better perceived latency. **How:** (1) OpenRouter (and most providers) support streaming chat completions. (2) In the pipeline, either yield tokens or return an iterator. (3) In FastAPI, return `StreamingResponse` with the token stream (e.g. SSE or newline-delimited JSON). (4) Streamlit chat can consume streaming responses if the client supports it (e.g. streamlit’s stream option or a custom component). **Link:** [app/main.py](../app/), [app/rag/pipeline.py](../app/), [frontend/chat_app.py](../frontend/chat_app.py). |
| **Feedback and abuse** | [app/main.py](../app/) (optional endpoint), [frontend/chat_app.py](../frontend/chat_app.py) | **What:** Thumbs up/down or report on answers; optional logging for abuse and model improvement. **How:** (1) Add `POST /api/feedback` with body like `{ "message_id": "...", "rating": 1\|-1, "comment": "..." }` and store in DB or append to a log. (2) Frontend: add buttons per assistant message and call the feedback API. (3) Optionally log query/response pairs (with PII stripped) for tuning and abuse detection. **Link:** [app/main.py](../app/), [frontend/chat_app.py](../frontend/chat_app.py). |

## Phase 4 — Scale and operations

| Item | Links | Deeper explanation |
|------|--------|---------------------|
| **Caching** | [app/rag/pipeline](../app/), [app/main.py](../app/) | **What:** Cache frequent (query, country) pairs to reduce LLM and retrieval load. **How:** In-memory (e.g. functools.lru_cache with a hash of query+country) or Redis. Cache key: normalized query + country; value: full response (or response + sources). Set TTL or max size to avoid stale data after index refresh. **Link:** [app/rag/pipeline.py](../app/) or a wrapper in [app/main.py](../app/). |
| **Rate limiting** | [app/main.py](../app/) | **What:** Per-API-key or per-IP rate limits on `/api/chat`. **How:** Use FastAPI middleware or a library (e.g. slowapi); limit by header `X-API-Key` or by request.client.host. Return 429 with Retry-After when exceeded. **Link:** [app/main.py](../app/), [requirements.txt](../requirements.txt) (add dependency if needed). |
| **Observability** | [app/main.py](../app/), [app/rag/](../app/) | **What:** Metrics (latency, error rate, retrieval top_k) and tracing for pipeline stages. **How:** (1) Add OpenTelemetry (or Prometheus) instrumentation: span per request, spans for retrieval vs LLM. (2) Log or export latency percentiles, error count, and optionally retrieval result counts. (3) Optionally integrate with Grafana/Datadog for dashboards. **Link:** [app/main.py](../app/), [app/rag/pipeline.py](../app/), [requirements.txt](../requirements.txt). |
| **Config and secrets** | [.env.example](../.env.example), [app/](../app/) | **What:** All config (model, weights, index path) via env or config file; secrets only from env/vault. **How:** (1) Extend [.env.example](../.env.example) with EMBEDDING_MODEL, VECTOR_WEIGHT, BM25_WEIGHT, INDEX_PATH, etc. (2) In app, load from os.environ or a small config module; never hardcode API keys or paths. (3) In Docker/Compose, pass env from host or secrets. **Link:** [.env.example](../.env.example), [app/](../app/). |

## Phase 5 — Advanced RAG and intelligence

| Item | Links | Deeper explanation |
|------|--------|---------------------|
| **Structured output** | [app/rag/pipeline](../app/), [app/main.py](../app/) | **What:** Optional JSON schema for answers (e.g. product list, price, policy summary) for downstream systems. **How:** (1) Add a query param or header like `Accept: application/json` or `response_format=structured`. (2) In the RAG prompt, ask the LLM to output JSON matching a schema (e.g. `{ "products": [...], "answer": "..." }`). (3) Parse and validate the model output; return structured body. **Link:** [app/rag/pipeline.py](../app/), [app/main.py](../app/). |
| **Multi-language** | [pipelines/indexing/build_vector_index.py](../pipelines/indexing/build_vector_index.py), [app/rag/](../app/), [.env.example](../.env.example) | **What:** Multilingual embedding and/or LLM for queries and answers in local languages. **How:** (1) Use a multilingual embedding model (e.g. paraphrase-multilingual-MiniLM) in [build_vector_index.py](../pipelines/indexing/build_vector_index.py) and in the retriever. (2) In the RAG prompt, add “Answer in the same language as the user’s query” or detect language and set response language. (3) Optionally use a multilingual LLM. **Link:** [build_vector_index.py](../pipelines/indexing/build_vector_index.py), [app/rag/](../app/), [.env.example](../.env.example). |
| **Agentic flows** | [app/rag/](../app/), new tools/actions | **What:** Optional tool use (e.g. check stock, place hold) with same guardrails and context. **How:** (1) Define tools (e.g. get_stock(product_id, country)) that call internal APIs or DBs. (2) Use an LLM that supports tool calls (e.g. OpenRouter function calling); pass context + allowed tools. (3) Execute only tools that do not expose internal data; apply same refusal logic to tool results before adding to context. **Link:** [app/rag/](../app/), new modules for tools and execution. |
